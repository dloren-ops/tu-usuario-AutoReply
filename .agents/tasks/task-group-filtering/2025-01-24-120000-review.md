# Per-rule group filtering via auto-discovered groups

This change lets users restrict individual auto-reply rules to specific WhatsApp groups (or groups from any monitored app). Groups are discovered automatically from incoming notifications and stored locally; the rule editor then shows checkboxes so users can pick which groups a rule applies to without typing names. If no groups are selected, the rule behaves identically to the prior version (replies everywhere its scope allows).

Watch for: **Race between upsert and ID lookup** (confirmed) — the group ID used for filtering may be null on first contact due to a fire-and-forget insert followed by a blocking read that could execute before the upsert completes. **`runBlocking` on the notification binder thread** (confirmed) — blocks a system thread on every group message, which could degrade notification delivery under load.

## High-level view

The data layer adds a `known_groups` table with a unique index on `conversationKey`, migrated cleanly from v4 to v5. The rule entity gains an `allowedGroupIds` column stored as a comma-separated string and deserialized to `Set<Long>` on read.

In the service layer, the notification listener registers groups on sight via an async upsert, then immediately queries the same repository for the group's database ID. That ID feeds into `ReplyEngine.groupAllowed()`, which gates rule evaluation. The logic is additive: it slots in after scope matching and before keyword matching, and short-circuits to `true` when the set is empty — preserving pre-existing behavior.

The UI surfaces the group list as scrollable checkboxes inside the rule editor dialog, with community-parent hints and multi-app disambiguation. Selection state is cleared when scope switches to `INDIVIDUAL_ONLY`.

The critical invariants are preserved: `ReplyGuard.kt` has zero diff lines, `conversationKey()` generation is unchanged, and the deduplication/signature pipeline remains untouched.

<summary>Issues (3)</summary>

1. **Upsert/read race on first group contact** — The `scope.launch` for upserting a new group is not awaited before the `runBlocking` read. On the very first notification from a new group, `getByConversationKey` may return null, causing `groupId` to be null and making the rule pass through unfiltered (fail-open). Replace the fire-and-forget launch with a direct `runBlocking` call or combine upsert+read into a single transaction returning the ID.
2. **`runBlocking` on system binder thread** — Every group notification blocks the notification binder thread while performing a Room query. Under burst traffic (community with many messages), this could cause Android to consider the listener unresponsive. Consider caching group IDs in memory (populated from the Flow) and falling back to a blocking read only on cache miss.
3. **Stale group references in rules** — Groups can be pruned via `deleteOlderThan`, but rules retain their IDs in `allowedGroupIds`. After pruning, a rule may reference a group ID that no longer exists, causing silent filter failure (the group won't match and the rule won't fire for that group even if it reappears with a new ID). Either skip pruning for groups referenced by rules, or cascade-clean rule references on prune.

<summary>Details</summary>

## Upsert-then-read race in the notification listener

The auto-discovery block launches the upsert as fire-and-forget (`scope.launch { ... upsert(...) }`) then immediately performs `runBlocking { knownGroupRepository.getByConversationKey(...) }`. On the first notification from a never-before-seen group, the upsert coroutine may not have reached the database yet when the read executes. The result is `groupId == null`.

Because `groupAllowed` treats a null `groupId` as "allow" (the comment says "no bloquear en primer contacto"), this is fail-open — the rule fires even if the group isn't in its allowed set. On subsequent messages the ID will exist and filtering will work correctly. So the window is one message wide per new group per app launch. Whether this matters depends on context: for most users it's fine, but if a rule is meant to silence a group entirely, the first message still triggers a reply.

The straightforward fix: replace the separate launch + runBlocking with a single `runBlocking { upsert(...); getByConversationKey(key)?.id }`. The `@Upsert` already handles conflict resolution, so ordering is guaranteed within a single coroutine.

## `runBlocking` on the notification binder thread

The notification listener's `onNotificationPosted` runs on a system binder thread. The `runBlocking` call dispatches to `Dispatchers.IO` (via Room's default) and suspends the binder thread until the query returns. Under normal traffic this is sub-millisecond, but in a high-volume community (dozens of messages per second), stacking blocked binder threads could trigger Android's watchdog for the listener service.

An in-memory cache of `conversationKey -> groupId`, populated from the reactive Flow already observed elsewhere, would eliminate the blocking read for all known groups. The blocking path would only trigger for brand-new groups (first message ever), which is acceptable.

## Stale group IDs after pruning

`KnownGroupDao.deleteOlderThan` allows removing groups that haven't been seen in a while. However, `RuleEntity.allowedGroupIds` stores raw IDs with no foreign-key relationship. If a group is pruned and later reappears, it gets a new auto-generated ID. Any rule that referenced the old ID will silently stop matching that group. There's no UI indication that a selected group no longer exists.

Options: (a) never auto-prune groups that are referenced by at least one rule, (b) show "unknown group" placeholders in the UI and let the user re-select, or (c) use `conversationKey` instead of numeric IDs as the stable identifier in `allowedGroupIds`.
